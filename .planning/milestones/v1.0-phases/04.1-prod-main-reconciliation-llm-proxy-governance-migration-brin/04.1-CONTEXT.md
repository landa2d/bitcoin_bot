# Phase 04.1: Prod↔Main Reconciliation + LLM-Proxy Governance Migration - Context

**Gathered:** 2026-05-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Bring production current with `main` and migrate the llm-proxy governance model
from file-based (`governance_config.json`) to DB-based (`agent_wallets_v2`),
establishing a clean, reconciled deploy baseline before any Phase 5+ feature code
ships to prod. Infrastructure/ops phase — the "users" are the operator and the
agents whose spending it governs.

**In scope:** the llm-proxy governance migration (data + code, atomic + fail-loud);
staged per-service deploy of the accumulated prod↔main code drift; deploy-baseline
guardrails.
**Out of scope:** new agent-economy features (Phases 5-10); the rivalscope negative-
balance anomaly (noted, deferred); deploying the new lab-data-provider container
(deferred — see below).
</domain>

<decisions>
## Implementation Decisions

### Canonical caps + downgrade behavior
- **D-01:** `governance_config.json` is the **canonical source** of per-agent cap
  values. Backfill `agent_wallets_v2` from the file exactly: analyst 28000/day,
  processor 1000/day, research 5000/week, newsletter 2000/week, gato 50000/day.
  (Resolves the live mismatches: analyst is 1000 in the DB vs 28000 in the file;
  processor/research/newsletter/gato are null in the DB.)
- **D-02:** gato's `downgrade_model` behavior (sonnet→deepseek on cap, instead of
  reject) **must survive** the migration. The DB governance model must be extended
  to represent per-agent on-cap behavior (reject vs downgrade + downgrade map), so
  cutover is zero behavior change.

### Fail-loud semantics
- **D-03:** A null/absent cap is **illegal**, never silently accepted. Every agent
  must carry either a numeric `spending_cap_sats` OR an explicit `uncapped=true`
  opt-in. Anything missing/ambiguous → **reject the call + emit a loud
  `governance_event`**. This is the structural fix for the silent-governance-off
  failure class ("the wallet bug all over again") — `proxy.py` must NOT fall through
  to `wallet.get("spending_cap_sats") is None → no enforcement`.
- **D-04:** Mechanism is the explicit-opt-in model (not a code allowlist, not
  refuse-to-boot). Planning decides whether `uncapped` is a new column/flag on
  `agent_wallets_v2` or an equivalent; the contract is "null is never a silent
  no-op."

### Deploy staging + verification
- **D-05:** Governance unit ships **first**, gated by a canary: after backfilling
  caps + deploying the new `proxy.py`, verify enforcement with a test call that MUST
  be capped/rejected (and a downgrade case for gato) BEFORE proceeding.
- **D-06:** After the governance unit verifies, deploy the remaining drifted services
  **one group at a time** (gato_brain, newsletter, processor, analyst, research, …),
  each via a scoped `rsync <dir>/ + docker compose build/up <svc>` — never a bare
  `scripts/deploy.sh --force-rebuild all`. Per-group rollback awareness.
- **D-07:** The new `lab-data-provider` container is **excluded** from this phase
  (deferred to a separate decision).

### Atomic-unit boundary + file retirement
- **D-08:** The atomic unit is: **backfill caps → deploy proxy.py → verify
  enforcement**. Atomicity across the non-transactional DB-change + code-deploy is
  enforced by ordering + the canary gate (caps land first; proxy.py only goes live
  after; enforcement is verified before any other service deploys).
- **D-09:** `governance_config.json` **stays in the repo** (and harmlessly on prod)
  as the cap source-of-record / audit trail after cutover — the new proxy ignores it.
  **No code fallback to the file** (a fallback would itself be a silent-default,
  violating D-03). It may be removed in a later cleanup, not on the critical path.

### Claude's Discretion
- Exact schema shape for `uncapped` + per-agent on-cap behavior (column vs JSON vs
  lookup) — planner/researcher decides, honoring D-02/D-03/D-04.
- Canary test mechanics and per-group deploy verification specifics.

### Folded Todos
- **Reconcile prod↔main drift and close the gap**
  (`.planning/todos/pending/2026-05-28-reconcile-prod-main-drift-and-close-the-gap.md`)
  — this phase IS the execution of that todo. The todo's audit findings, classified
  inventory, and the already-done repo-side cleanup (commit `5f6fadc`) are the
  starting baseline. Mark the todo resolved when this phase completes.
</decisions>
