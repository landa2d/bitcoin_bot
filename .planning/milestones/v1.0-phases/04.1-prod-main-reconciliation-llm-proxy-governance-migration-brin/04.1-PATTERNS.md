## Pattern Assignments

### 1. `docker/llm-proxy/proxy.py` (MODIFY — middleware, request-response)

**Analog:** itself. The governance code already lives here and is DB-based. The phase
*extends* three points; it does not rewrite the service.

#### ⚠️ Load-bearing finding #1 — the silent-no-op (D-03 fix target)

The exact fall-through the operator named "the wallet bug all over again" is the
truthiness guard at **line 351**. A `None` (absent) or `0` cap silently skips the
entire cap block → call proceeds with **no enforcement**. This is the line D-03 makes illegal.

`check_governance()` — `docker/llm-proxy/proxy.py:323-365`:
```python
def check_governance(agent_name: str, model: str, balance_after: int) -> dict:
    """
    Run governance checks AFTER reserve_balance succeeds.
    Returns dict with 'action' key: 'allow', 'reject', or 'downgrade'.   # <-- docstring lies (see finding #2)
    Governance events fire regardless of action.
    """
    wallet = _get_wallet(agent_name)
    if not wallet:
        return {"action": "allow"}                       # <-- D-03: unknown agent also silently allowed
    ...
    # --- Spending cap check ---
    cap_sats = wallet.get("spending_cap_sats")
    cap_window = wallet.get("spending_cap_window", "daily")
    if cap_sats and cap_sats > 0:                         # <-- LINE 351: THE SILENT NO-OP. null/0 → no enforcement
        window_total = _get_spending_window_total(agent_name, cap_window)
        if window_total >= cap_sats:
            _emit_governance_event(agent_name, "cap_hit", {
                "spending_cap_sats": cap_sats,
                "window": cap_window,
                "window_total_sats": window_total,
                "model": model,
            })
            # For internal agents (allow_negative), alert but don't reject
            if not wallet.get("allow_negative", False):
                return {"action": "reject", "reason": f"Spending cap exceeded ({window_total}/{cap_sats} sats in {cap_window} window)"}
            # Internal agents: log but allow through

    return {"action": "allow"}
```

**Required change (D-03/D-04):** Replace `if cap_sats and cap_sats > 0:` with an
explicit three-way contract:
- numeric `spending_cap_sats > 0` → enforce as today
- explicit `uncapped == true` (the new opt-in flag from migration 034) → allow, no event
- **everything else** (null/absent/ambiguous AND not uncapped) → `{"action": "reject", ...}`
  + emit a **loud** `governance_event` (the `_emit_governance_event` helper below). Never fall through to allow.
- Also harden the `if not wallet: return {"action": "allow"}` at line 330-331 — an
  unknown agent is the same illegal-ambiguous case under D-03 (reject + emit), not silent allow.

#### ⚠️ Load-bearing finding #2 — `downgrade` is advertised but NOT implemented (D-02 fix target)

The docstring (line 326) promises a `'downgrade'` action, but `check_governance`
**only ever returns `'allow'` or `'reject'`** — there is no code path that returns
`'downgrade'`, and neither call site handles it. gato's sonnet→deepseek downgrade
currently lives ONLY in `governance_config.json` (`"on_cap_exceeded": "downgrade_model"`,
`"downgrade_map": {...}`), which the main-branch `proxy.py` **does not read**. So at
cutover, gato would start getting `reject` instead of a model downgrade unless this is built.

Both call sites only handle `reject` — `docker/llm-proxy/proxy.py:760-769` (chat) and `:959-964` (the second/embeddings path):
```python
    # Governance checks (spending caps, balance thresholds)
    # Skip governance if balance is None (RPC failed — no reliable balance data)
    if balance is not None:
        gov = check_governance(agent_name, model, balance - estimated_sats)
        if gov["action"] == "reject":                                    # <-- only 'reject' handled
            settle_balance(agent_name, estimated_sats, 0)  # refund the reservation
            return JSONResponse(
                status_code=429,
                content={"error": {"message": gov.get("reason", "Governance limit reached"), "type": "governance_error"}},
            )
    # ...no `elif gov["action"] == "downgrade"` branch exists...
```

**Required change (D-02):** Represent per-agent on-cap behavior in `agent_wallets_v2`
(reject vs downgrade + downgrade map — schema shape is Claude's discretion per D-04),
have `check_governance` return `{"action": "downgrade", "new_model": <mapped>}` for
gato, and add a `downgrade` branch at BOTH call sites. The downgrade must rewrite the
locally-held `model`, re-resolve `route`, and patch `body["model"]` **before** the
upstream request is built — see the rebindable locals at the call site:

`docker/llm-proxy/proxy.py:727-733` (model/route are local and rewritable):
```python
    model = body.get("model", "")
    route = MODEL_ROUTES.get(model)
    if not route:
        return JSONResponse(
            status_code=400,
            content={"error": {"message": f"Unknown model: {model}. Available: {list(MODEL_ROUTES.keys())}", "type": "invalid_request"}},
        )
```
A downgrade branch must set `model = new_model; route = MODEL_ROUTES.get(new_model); body["model"] = new_model`
and re-run the allowed-models/provider-key resolution before reserving + sending upstream.
Note the ordering wrinkle: `check_governance` runs AFTER `reserve_balance` (line 753 → 763),
so a downgrade after reservation should re-estimate cost / re-reserve, or downgrade
must be decided pre-reserve. Planner picks; flag it.

#### Reusable helper — the loud governance-event emitter (use for D-03 rejects)

`_emit_governance_event()` — `docker/llm-proxy/proxy.py:265-280`. This is the exact
mechanism for the "emit a loud `governance_event`" requirement. Note the 5-minute
dedup cooldown (`GOV_EVENT_COOLDOWN = 300.0`, line 137) — a brand-new event_type for
the illegal-cap case (e.g. `"cap_missing"` / `"governance_misconfig"`) is fine, but
that string must be added to the `governance_events.event_type` CHECK constraint (see
migration analog below — 023 line 69 has a closed CHECK list that does NOT currently
include a missing-cap type):
```python
def _emit_governance_event(agent_name: str, event_type: str, details: dict):
    """Log a governance event (with dedup cooldown)."""
    key = f"{agent_name}:{event_type}"
    now = time.time()
    if now - _gov_event_sent.get(key, 0) < GOV_EVENT_COOLDOWN:
        return  # already emitted recently
    _gov_event_sent[key] = now
    try:
        _sb_insert("governance_events", {
            "agent_name": agent_name,
            "event_type": event_type,
            "details": details,
        })
        logger.info(f"[GOVERNANCE] {event_type} for {agent_name}: {details}")
    except Exception as e:
        logger.warning(f"Failed to log governance event {event_type} for {agent_name}: {e}")
```

#### DB read path (D-09 — already DB-only, confirm no file fallback exists)

The wallet (and thus governance config) is read from `agent_wallets_v2` with a 15s
cache. `proxy.py` contains **zero references to `governance_config.json`** (grep
confirmed) — the main-branch proxy is already DB-only, so D-09 ("no code fallback to
the file") is *already satisfied in this code*; the gap is purely the new columns +
data, not removing a fallback. Do NOT add a file read.

`_get_wallet()` — `docker/llm-proxy/proxy.py:249-262`:
```python
def _get_wallet(agent_name: str) -> dict | None:
    """Get wallet record with caching."""
    now = time.time()
    if agent_name in wallet_cache and now - wallet_cache_ts.get(agent_name, 0) < WALLET_CACHE_TTL:
        return wallet_cache[agent_name]
    try:
        result = supabase.table("agent_wallets_v2").select("*").eq("agent_name", agent_name).execute()
        if result.data:
            wallet_cache[agent_name] = result.data[0]
            wallet_cache_ts[agent_name] = now
            return result.data[0]
    except Exception as e:
        logger.warning(f"Failed to fetch wallet for {agent_name}: {e}")
    return wallet_cache.get(agent_name)
```
Because `select("*")` is used, the new `uncapped` / on-cap-behavior columns from
migration 034 flow through to `wallet` automatically — no select-list change needed.

#### Repo idioms to reuse (Supabase helpers)

`docker/llm-proxy/proxy.py:164-180` — thin sync wrappers; reuse rather than calling
`supabase.*` inline:
```python
def _sb_rpc(fn_name: str, params: dict) -> Any:
    result = supabase.rpc(fn_name, params).execute()
    return result.data

def _sb_insert(table: str, record: dict):
    supabase.table(table).insert(record).execute()

def _sb_update(table: str, data: dict, match: dict):
    q = supabase.table(table).update(data)
    for k, v in match.items():
        q = q.eq(k, v)
    q.execute()
```
Conventions (per CLAUDE.md): broad `except Exception:` + `logger.warning/error`,
logger name `"llm-proxy"`, module-level globals, snake_case. The new logic should
match this — BUT D-03 explicitly overrides the "swallow + continue" habit for the
*enforcement decision itself*: a missing cap must hard-reject, not be swallowed.

#### Idempotent ADD COLUMN idiom (idempotency posture)

`033` ADDs a constraint guarded by an existence check (`033:120-138`). Postgres also
supports `ADD COLUMN IF NOT EXISTS` directly, which is the simpler idiom for a column add:
```sql
ALTER TABLE agent_wallets_v2
    ADD COLUMN IF NOT EXISTS uncapped BOOLEAN NOT NULL DEFAULT FALSE;
-- on-cap behavior (D-02/D-04): either dedicated columns ...
ALTER TABLE agent_wallets_v2
    ADD COLUMN IF NOT EXISTS on_cap_behavior TEXT NOT NULL DEFAULT 'reject'
        CHECK (on_cap_behavior IN ('reject','downgrade'));
ALTER TABLE agent_wallets_v2
    ADD COLUMN IF NOT EXISTS downgrade_map JSONB NOT NULL DEFAULT '{}'::jsonb;
-- ... OR a single JSONB governance blob. Shape is Claude's discretion per D-04.
```
(Repo precedent for `text + CHECK (col IN (...))` enums is noted at `033:30` /
`023:31,39-42,69-72`. Repo precedent for `JSONB DEFAULT '{}'::jsonb` is `023:16,49`
and the `governance_events.details` column.)

#### Idempotent data backfill idiom

`023` uses raw `CREATE FUNCTION` + table DDL; the idempotent *data* idiom in this repo
is `INSERT ... ON CONFLICT (...) DO NOTHING` (033:402-415, 020:18-25/39-45). But this
migration must **update existing rows** (analyst is currently `1000` in DB and must
become `28000`; others are null) — so it is an `UPDATE` / `ON CONFLICT DO UPDATE`,
not `DO NOTHING`. `033:280-328` shows in-repo `UPDATE <table> SET ... WHERE ...`
precedent. Recommended explicit per-agent UPDATEs (rows already exist on prod per the
audit), e.g.:
```sql
-- D-01: backfill exact canonical caps from governance_config.json
UPDATE agent_wallets_v2 SET spending_cap_sats = 28000, spending_cap_window = 'daily',
    on_cap_behavior = 'reject',    uncapped = FALSE WHERE agent_name = 'analyst';
UPDATE agent_wallets_v2 SET spending_cap_sats = 1000,  spending_cap_window = 'daily',
    on_cap_behavior = 'reject',    uncapped = FALSE WHERE agent_name = 'processor';
UPDATE agent_wallets_v2 SET spending_cap_sats = 5000,  spending_cap_window = 'weekly',
    on_cap_behavior = 'reject',    uncapped = FALSE WHERE agent_name = 'research';
UPDATE agent_wallets_v2 SET spending_cap_sats = 2000,  spending_cap_window = 'weekly',
    on_cap_behavior = 'reject',    uncapped = FALSE WHERE agent_name = 'newsletter';
UPDATE agent_wallets_v2 SET spending_cap_sats = 50000, spending_cap_window = 'daily',
    on_cap_behavior = 'downgrade', uncapped = FALSE,
    downgrade_map = '{"claude-sonnet-4-20250514": "deepseek-chat"}'::jsonb
    WHERE agent_name = 'gato';
```
Decision the planner MUST make: the audit lists prod agents with **null** caps that
are NOT in governance_config.json (`lab_data-provider 5000/d`, `rivalscope 10000/d`,
all `code_*`, most `lab_*`). Under D-03 every such agent must get either a cap or
`uncapped = TRUE` BEFORE the new proxy goes live, or its next call hard-rejects. The
migration (or a companion data step) must set `uncapped = TRUE` for the agents the
operator intends to leave uncapped (rivalscope is deferred but still needs a non-null
posture). This is the "every agent carries a cap OR explicit uncapped" sweep from D-03/D-04.

#### ⚠️ CHECK-constraint coupling for the new fail-loud event type

If `proxy.py` emits a new `event_type` (e.g. `'cap_missing'`) for the illegal-null
case, the `governance_events.event_type` CHECK at `023:69-72` is a **closed list** and
will reject the insert (and `_emit_governance_event` swallows the exception → silent!).
The migration must `ALTER` that CHECK to include the new type, OR reuse an existing
allowed type. Existing allowed types (`023:69-72`):
`'cap_hit','model_downgrade','rate_limit','balance_low','balance_exhausted','fallback_triggered'`.
(`020`'s `governance_events` has NO check on event_type — but `023` is the one on prod.)
Reusing `'cap_hit'` with a distinguishing `details` payload avoids the constraint edit;
adding a new type is cleaner but couples migration↔code. Planner picks; this coupling
is load-bearing for fail-loud actually working.

## Shared Patterns

### Fail-loud governance (D-03 spine) — applies to all proxy.py governance edits
**Source pattern:** `_emit_governance_event` (`proxy.py:265-280`) + hard `reject`
return (`proxy.py:362`). **Apply to:** the rewritten cap-check and the unknown-agent
branch. Contract: any null/absent/ambiguous cap that is NOT `uncapped=true` →
`{"action": "reject"}` + a loud governance_event. This OVERRIDES the repo's habitual
"`except Exception:` → log → continue" for the enforcement *decision* (CLAUDE.md memory:
"Fail-loud governance / the wallet bug all over again").

### Structural over application enforcement (memory preference)
**Apply to:** prefer expressing the cap/uncapped contract in the schema (NOT NULL +
CHECK that cap_sats IS NOT NULL OR uncapped) so the DB itself forbids the ambiguous
state, rather than relying solely on the proxy. A `CHECK (spending_cap_sats IS NOT NULL OR uncapped)`
on `agent_wallets_v2` would make the illegal state unrepresentable. Planner's discretion
(D-04) but aligns with the operator's stated preference for triggers/CHECK over app checks.

### Supabase access idioms — applies to migration + proxy
**Source:** `proxy.py:164-180` (`_sb_rpc`/`_sb_insert`/`_sb_update`), `select("*")` reads
(`proxy.py:255`). No ORM, module-level global `supabase` client, raw SQL migrations
applied via MCP/manual (CLAUDE.md). The `select("*")` read means new columns surface
to the proxy with no select-list edit.

### Idempotent migration posture — applies to 034
**Source:** `IF NOT EXISTS` (020 throughout, 033:121-137), `ON CONFLICT DO NOTHING`
for pure seeds (033:415, 020:25/45). For 034 use `ADD COLUMN IF NOT EXISTS` + explicit
`UPDATE`/`DO UPDATE` (because rows exist and values must change — D-01 corrects the
analyst 1000→28000 mismatch). Re-runnable without clobbering operator intent.
