## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator approval -> batch publish (service_role write) | The batch script writes LIVE prod content under the service_role key; the single operator-approval gate is the trust boundary before any RPC fires. |
| anon visitor -> economy_map (verification read) | The verification harness reads with the anon key only, to prove what a real visitor (no privilege) sees post-publish. |
| repo source -> committed secrets | Both scripts load keys from config/.env at runtime; a hardcoded key would leak into git. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-18-05 | Elevation of Privilege | service_role key in the verification path | mitigate | verify_economy_map_publish.py reads with SUPABASE_ANON_KEY ONLY (grep-asserted) — it never loads or uses the service_role key, so the visitor-perspective proof cannot mask an RLS gap. |
| T-18-06 | Information Disclosure | service_role key hardcoded in either script | mitigate | Both scripts load keys from config/.env via _load_env_file (copied from the Phase-16/17 idiom); no key literal in source — the existing repo-wide leak guard (verify_economy_map_crosslinks.py --guard-only) remains the gate (run in plan 03). |
| T-18-07 | Denial of Service | partial-pass on mid-batch failure | mitigate | Fail-loud halt-and-report: any non-idempotent RPC error HALTs and prints succeeded/remaining; a re-run completes the halted batch idempotently (D-08, fail_loud_governance). |
| T-18-SC | Tampering | new package installs | accept | No new packages — both scripts import only stdlib + the already-vendored httpx (no pip install task; no Package Legitimacy Gate needed). |
