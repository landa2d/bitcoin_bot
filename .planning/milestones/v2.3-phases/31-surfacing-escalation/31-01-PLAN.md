<objective>
SURF-01: harden the processor's `send_telegram` so a held/escalated edition can never silently fail to alert. Today it does a bare `return` when `TELEGRAM_BOT_TOKEN`/`TELEGRAM_OWNER_ID` are unset (silent no-op) and returns `None` on delivery failure. This plan hardens it IN PLACE (D-01 — no shared/copied helper, no cross-service routing; the newsletter's `_alert_operator` stays untouched): make it return a bool, ERROR-log (never `warning`, never bare return) on both env-unset and send failure, add a boot-time grep-able ERROR when the env is unset, and make the hold/eval-critical auto-publish caller check the return.

Purpose: the eval is LIVE report-only (armed 2026-07-02) and its hold/escalation alerts route through `send_telegram`; a silent alert loss during the calibration window is the exact failure mode this milestone fights ("the wallet bug all over again"). Loud-on-failure makes an alerting gap unmissable without ever crashing ingestion.
Output: a bool-returning, fail-loud `send_telegram`; a startup config ERROR; a critical-caller return check on the auto-publish notification; a unit test suite proving the contract.
</objective>
