## Scope note (why PASS despite §2/§3 not driven live)

The cutover's stated goals — **fail-loud on MISSING caps** and **zero-behavior-change** — are met and verified. The over-cap behavior is *soft* for all five governed agents because `allow_negative=true` (a pre-existing setting the cutover deliberately preserved; not a regression — confirmed identical at `cbdaa1d^`). The operator explicitly accepted soft caps as the intended model: caps are monitoring/alerting, and the hard guarantee is fail-loud-on-missing (verified). The §2.1 hard-reject and §3 downgrade tests were therefore not applicable to live data and were not forced (doing so would mean real spend + halting real agent work).

**Follow-up (not a blocker for this phase):** if hard cap enforcement is later desired, set `agent_wallets_v2.allow_negative=false` for the capped agents (balance-reservation tolerance is driven separately by `access_tier=="internal"`, so the column flip only hardens `check_governance`).
