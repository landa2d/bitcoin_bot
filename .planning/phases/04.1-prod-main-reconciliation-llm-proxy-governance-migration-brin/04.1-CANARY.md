# 04.1 Canary Procedure — LLM-Proxy Governance Cutover

**Owner:** operator (executed by Plan 04.1-02, NOT by Plan 04.1-01)
**Runs:** AFTER migration `034_governance_caps_and_oncap_behavior.sql` is applied to live
Supabase AND AFTER the new `proxy.py` is deployed via scoped `--force-rebuild llm-proxy`,
and BEFORE any other service deploys (D-05 / D-08).

This is a deterministic PASS/FAIL gate. Every step has an exact command and an exact
expected result. Do not proceed to the staged per-service deploy (Plan 04.1-03) until
ALL of sections 1–4 PASS.

---

## Timing caveats (read first — prevents misreading a cached/deduped result)

- `WALLET_CACHE_TTL = 15s` (proxy.py): the proxy caches each agent's wallet row for 15
  seconds. After applying the migration, wait **>15s** before the first canary call, or
  the proxy may still serve the pre-migration (null-cap) wallet from cache and reject for
  the wrong reason. Likewise after any DB edit made during the canary, wait >15s.
- `GOV_EVENT_COOLDOWN = 300s` (proxy.py): `_emit_governance_event` dedups the same
  `agent:event_type` for 5 minutes. If you drive the same agent over cap twice within 5
  minutes you will see ONE governance_event row, not two — this is expected, not a miss.
  When asserting an event row, look back ~5 minutes and accept exactly one.
- The proxy container is `agentpulse-llm-proxy`, service name `llm-proxy`, bound to
  `127.0.0.1:8200` on the prod host. Canary HTTP calls run ON the prod host (or via an
  ssh tunnel) against `http://127.0.0.1:8200`.
- Agent API keys follow `ap_<agent>_<hash>`. Use each agent's real key from the registry
  for the per-agent calls below.

## SECTION 1 — DB-STATE PROOF (post-migration)

Run via Supabase SQL (MCP read / psql). Assert the exact caps, gato's downgrade config,
and the D-03 zero-ambiguous-rows invariant.

### 1.1 Exact caps + gato downgrade map

```sql
SELECT agent_name, spending_cap_sats, spending_cap_window, on_cap_behavior, uncapped, downgrade_map
FROM agent_wallets_v2
WHERE agent_name IN ('analyst','processor','research','newsletter','gato')
ORDER BY agent_name;
```

EXPECTED (exactly these rows):

| agent_name | spending_cap_sats | spending_cap_window | on_cap_behavior | uncapped | downgrade_map |
|------------|-------------------|---------------------|-----------------|----------|---------------|
| analyst    | 28000 | daily  | reject    | false | {} |
| gato       | 50000 | daily  | downgrade | false | {"claude-sonnet-4-20250514": "deepseek-chat"} |
| newsletter | 2000  | weekly | reject    | false | {} |
| processor  | 1000  | daily  | reject    | false | {} |
| research   | 5000  | weekly | reject    | false | {} |

PASS only if all five rows match exactly (note analyst MUST be 28000, not the pre-migration 1000).

### 1.2 D-03 invariant — zero rows left in the illegal ambiguous state

```sql
SELECT count(*) AS ambiguous_rows
FROM agent_wallets_v2
WHERE spending_cap_sats IS NULL AND uncapped = FALSE;
```

EXPECTED: `ambiguous_rows = 0`. PASS only if 0. (Any nonzero count means an agent — e.g.
a new `code_*`/`lab_*` or rivalscope — was left with a null cap and not marked uncapped,
which the new proxy will hard-reject on its next call.)

### 1.3 Sweep proof — the non-capped agents are explicitly uncapped

```sql
SELECT agent_name, spending_cap_sats, uncapped
FROM agent_wallets_v2
WHERE agent_name NOT IN ('analyst','processor','research','newsletter','gato')
ORDER BY agent_name;
```

EXPECTED: every returned row has either a positive `spending_cap_sats` OR `uncapped = true`
(rivalscope, lab_data-provider, all code_*/lab_*). PASS only if none has null-cap-and-not-uncapped.

## SECTION 2 — FAIL-LOUD CANARY (D-03)

### 2.1 Capped agent over its window → 429 governance_error

Pick a capped agent with the smallest window (processor, 1000/daily) so it is cheapest to
push over. Drive it over its daily window (either it is already over from real traffic, or
issue enough small deepseek-chat calls to exceed 1000 sats in the current day). Then issue
one more call:

```bash
curl -s -o /tmp/canary_reject.json -w "%{http_code}\n" \
  -X POST http://127.0.0.1:8200/v1/chat/completions \
  -H "Authorization: Bearer ap_processor_<hash>" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"canary"}]}'
cat /tmp/canary_reject.json
```

EXPECTED: HTTP `429`, body `{"error":{"message":"Spending cap exceeded (...)","type":"governance_error"}}`.
PASS only if 429 + `"type":"governance_error"`. (A 200 here means the cap is not enforced — FAIL.)

### 2.2 Confirm the loud governance_event row was written

```sql
SELECT agent_name, event_type, details, created_at
FROM governance_events
WHERE agent_name = 'processor'
  AND created_at > NOW() - INTERVAL '5 minutes'
ORDER BY created_at DESC;
```

EXPECTED: at least one row with `event_type = 'cap_hit'` for the over-cap reject (and, if you
exercised a missing-cap/unknown agent below, a `cap_missing` row). PASS only if the row exists.
(Remember GOV_EVENT_COOLDOWN=300s — exactly one row per type per 5-minute window is expected.)

### 2.3 Negative test — unknown agent → reject (NOT allow)

Temporarily call with a syntactically valid key for an agent that has NO `agent_wallets_v2`
row (or use a registry agent you have deliberately left without a wallet row in a staging
check). Confirm the proxy rejects rather than silently allowing:

```bash
curl -s -o /tmp/canary_unknown.json -w "%{http_code}\n" \
  -X POST http://127.0.0.1:8200/v1/chat/completions \
  -H "Authorization: Bearer ap_<unknown_agent>_<hash>" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"canary"}]}'
cat /tmp/canary_unknown.json
```

EXPECTED: HTTP `429` with `"type":"governance_error"` and a reason mentioning "No
wallet/governance record", AND a `governance_events` row with `event_type = 'cap_missing'`
for that agent. PASS only if reject + the `cap_missing` event. (Authentication must succeed
first — an unknown *key* yields 401 auth_error, which is a different path; this test needs a
valid key for an agent with no wallet row.)

```sql
SELECT agent_name, event_type, details, created_at
FROM governance_events
WHERE event_type = 'cap_missing'
  AND created_at > NOW() - INTERVAL '5 minutes'
ORDER BY created_at DESC;
```

EXPECTED: a `cap_missing` row exists. PASS only if present (proves the migration's CHECK
extension accepted the new event_type — the insert did NOT get silently dropped).

## SECTION 3 — DOWNGRADE CANARY (D-02, gato)

gato's on-cap behavior is `downgrade` (sonnet → deepseek). The downgrade is **cross-provider**
(anthropic → deepseek), so the proxy's anthropic handler does NOT silently serve sonnet and
does NOT plainly reject — it returns a **typed 429 `governance_downgrade`** redirect pointing
the caller at the chat endpoint, and emits a `model_downgrade` governance_event. This is the
mechanism chosen in Plan 04.1-01 Task 2 (option (ii): typed redirect, no anthropic→OpenAI body
translation).

### 3.1 Drive gato over its 50000/daily cap with a sonnet request on the anthropic endpoint

Ensure gato's daily window is ≥50000 sats (from real traffic or by driving calls), then:

```bash
curl -s -o /tmp/canary_downgrade.json -w "%{http_code}\n" \
  -X POST http://127.0.0.1:8200/anthropic/v1/messages \
  -H "x-api-key: ap_gato_<hash>" \
  -H "anthropic-version: 2023-06-01" \
  -H "Content-Type: application/json" \
  -d '{"model":"claude-sonnet-4-20250514","max_tokens":64,"messages":[{"role":"user","content":"canary"}]}'
cat /tmp/canary_downgrade.json
```

EXPECTED: HTTP `429` with body
`{"error":{"message":"Spending cap exceeded (...) — downgraded claude-sonnet-4-20250514 -> deepseek-chat","type":"governance_downgrade","new_model":"deepseek-chat","retry_endpoint":"/v1/chat/completions"}}`.

PASS only if `"type":"governance_downgrade"` AND `"new_model":"deepseek-chat"` AND
`"retry_endpoint":"/v1/chat/completions"`.
FAIL if it is a 200 sonnet response (silent pass-through) or a plain `governance_error` reject.

### 3.2 Confirm the model_downgrade governance_event row

```sql
SELECT agent_name, event_type, details, created_at
FROM governance_events
WHERE agent_name = 'gato'
  AND event_type = 'model_downgrade'
  AND created_at > NOW() - INTERVAL '5 minutes'
ORDER BY created_at DESC;
```

EXPECTED: one row with `details` containing `from_model = claude-sonnet-4-20250514` and
`to_model = deepseek-chat`. PASS only if present.

### 3.3 (Optional confirmation) the redirected model serves on the chat endpoint

Following the redirect, the same gato key calling `deepseek-chat` on the chat endpoint should
be served normally (deepseek-chat is in gato's allowed_models):

```bash
curl -s -o /tmp/canary_downgrade_retry.json -w "%{http_code}\n" \
  -X POST http://127.0.0.1:8200/v1/chat/completions \
  -H "Authorization: Bearer ap_gato_<hash>" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"canary"}]}'
cat /tmp/canary_downgrade_retry.json
```

EXPECTED: HTTP `200` (deepseek-chat is not over its own constraint here — gato's cap is the
daily window total; if the window is still ≥50000 this deepseek call may also redirect, which
is acceptable). This step is confirmatory, not part of the binary gate.

## SECTION 4 — ZERO-BEHAVIOR-CHANGE CHECK (under-cap, all five agents)

For each of the five capped agents, an UNDER-cap call with an allowed model must return 200
and be neither rejected nor downgraded. Run each with the agent's real key and an allowed
model (analyst/processor → deepseek-chat; research → claude-sonnet-4-20250514 on the anthropic
endpoint; newsletter → gpt-4o-mini or deepseek-chat; gato under cap → claude-sonnet-4-20250514
on the anthropic endpoint). Example (processor):

```bash
curl -s -o /dev/null -w "processor %{http_code}\n" \
  -X POST http://127.0.0.1:8200/v1/chat/completions \
  -H "Authorization: Bearer ap_processor_<hash>" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"canary"}]}'
```

EXPECTED for each of analyst, processor, research, newsletter, gato (when under its window):
HTTP `200`. PASS only if all five return 200 under cap. Also confirm the swept uncapped agents
still serve — pick one previously-null agent (e.g. lab_data-provider) and confirm a 200:

```bash
curl -s -o /dev/null -w "lab_data-provider %{http_code}\n" \
  -X POST http://127.0.0.1:8200/v1/chat/completions \
  -H "Authorization: Bearer ap_lab_data-provider_<hash>" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"canary"}]}'
```

EXPECTED: HTTP `200` (the uncapped sweep marked it uncapped=TRUE, so it is allowed, not
hard-rejected). PASS only if 200.

### Rollback note

- The migration (034) is additive + idempotent and is SAFE TO LEAVE in place on rollback —
  the new columns, the CHECK, and the event_type extension do not break the prior file-based
  proxy. The only data change that *could* be reverted is the uncapped sweep (re-nulling the
  swept agents), but reverting it is unnecessary and not recommended.
- The actual rollback lever is the PROXY IMAGE: roll `proxy.py` back to the prior deployed
  image via `scripts/deploy.sh --force-rebuild llm-proxy` against the previous commit. The
  prior file-based proxy resumes serving (it reads governance_config.json, ignores the new
  columns). Because the migration is left in place, a subsequent re-deploy of the new proxy
  needs no DB re-apply.
- After rollback, re-investigate the failing canary section before re-attempting cutover.
  Do NOT deploy any other drifted service while the governance unit is in a failed/rolled-back
  state (D-08 ordering: governance verifies FIRST).
