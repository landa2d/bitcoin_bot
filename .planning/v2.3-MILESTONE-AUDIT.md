# Milestone v2.3 Audit — Pre-Publish Evaluation Step

**Status: gaps_found** — one root-cause production blocker, discovered by the cross-phase integration check.

## Headline

Every phase (26–31) individually verified `passed`, and every module is correct and unit-tested at the
source level. But `docker/newsletter/Dockerfile` was never updated across Phases 27/28/29 to COPY the
three new modules (`edition_eval.py`, `deterministic_gate.py`, `judge_loop.py`) into the image. The
deployed newsletter container therefore cannot import the eval pipeline: **every eval invocation since
report-only arming (2026-07-02) has crashed with `ModuleNotFoundError`**, silently (the outer catches
log ERROR but never alert the operator). Live proof: edition #104 (generated 2026-07-03, post-arming)
crashed on both save points; `edition_evals` holds 0 rows.

Why no gate caught it: the test suite imports the modules from the git source tree (`conftest.py`
inserts `docker/newsletter/` on `sys.path`), so packaging is invisible to tests; 30-04's deploy
verification grepped `run_edition_eval` inside `newsletter_poller.py` (present) but never resolved the
lazy imports inside the running container; the 27-03/30-04 "settled governed call" proofs were manual
`httpx` calls against the proxy, bypassing the real sequencer path. Phase 31's D-14 ("never rebuild
newsletter during calibration") then locked the broken image in place.
